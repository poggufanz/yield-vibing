## 2. User Stories per Feature

### Feature: Venice AI Strategy + Skill Generation

**US-01**
> As a yield farmer, I want to input my desired USDC amount, risk level, and number of vaults, so that Venice AI can automatically generate a multi-vault strategy and skill sets for each agent.

**Acceptance Criteria:**
- [ ] Input form is available: USDC amount (numeric), risk level (Low/Medium/High), and number of vaults (integer, min 2).
- [ ] Click on the "Generate Strategy" button sends a request to Venice AI.
- [ ] Response displays: allocation per vault (name, address, amount, APY).
- [ ] Response displays a skill card per agent: swap parameters + deposit parameters.
- [ ] Display privacy badge: "Coordinated by Venice AI — No data retention".
- [ ] Loading state is displayed while waiting for the response (max 10 seconds).
- [ ] Error state + hardcoded fallback strategy if Venice AI does not respond.

**US-02**
> As a yield farmer, I want to understand why Venice AI recommends this strategy, so that I can make an informed decision before approving it.

**Acceptance Criteria:**
- [ ] Brief reasoning is displayed per vault (≤ 2 sentences).
- [ ] Estimated APY per vault is displayed.
- [ ] Total expected APY (weighted average) is displayed in the summary card.

---

### Feature: Skill Review + Edit UI

**US-03**
> As a yield farmer, I want to view and edit the skill sets generated by Venice AI for each agent, so that I retain full control over the execution parameters.

**Acceptance Criteria:**
- [ ] A skill card is displayed per agent (one card per Worker Agent).
- [ ] Each card displays: `agentId`, target vault, `maxSlippage`, `dexPreference`, and `maxAmount`.
- [ ] User can edit: `maxSlippage` (input range 0–2%), `maxRetries` (dropdown 1–3).
- [ ] Clicking the "Approve Skill Sets" button locks all skills and writes them to files.
- [ ] Execution cannot begin before approval is completed.

**US-04**
> As a yield farmer, I want to know what a "skill" means for this agent, so that I do not approve parameters I do not understand.

**Acceptance Criteria:**
- [ ] Tooltips are available per skill parameter explaining its meaning in plain terms.
- [ ] Example: "maxSlippage: the maximum price change percentage the agent will tolerate during a swap."

---

### Feature: Wallet Connect + EIP-7702

**US-05**
> As a user, I want to connect MetaMask Flask and upgrade my account to a smart account, so that agents can execute transactions within the specific permission limits I configure.

**Acceptance Criteria:**
- [ ] "Connect Wallet" button detects the MetaMask Flask extension.
- [ ] If MetaMask Flask is not found: display the warning message "Install MetaMask Flask (not regular MetaMask)".
- [ ] Upon connection: display the account address (shortened format: 0x1234...5678).
- [ ] The application detects whether the account is already a smart account or still an EOA.
- [ ] If EOA: display instructions + trigger the EIP-7702 authorization.
- [ ] Upon successful upgrade: display the badge "Smart Account Active".
- [ ] Sepolia network is detected and shown.

---

### Feature: ERC-7715 Permission per Agent

**US-06**
> As a yield farmer, I want to set different permissions with clear limits for each agent, so that each agent can only act on the vault I allow with the amount I specify.

**Acceptance Criteria:**
- [ ] A permission card is displayed per agent (matching the approved skills).
- [ ] Each card displays: `agentId`, vault address, `maxAmount`, and expiration time.
- [ ] Clicking "Grant Permissions for All Agents" triggers ERC-7715 permission request per agent.
- [ ] MetaMask Flask displays a permission dialog per agent (or batched).
- [ ] After all are approved: the UI updates to "2 Agent Permissions Active".
- [ ] A "Revoke All Permissions" button becomes available after granting.

**US-07**
> As a yield farmer, I want to be able to revoke permissions for all agents at any time, so that I maintain absolute control even after the agents have been launched.

**Acceptance Criteria:**
- [ ] A "Revoke All Permissions" button is available once permissions are active.
- [ ] After revocation: status updates to "No Active Permissions".
- [ ] Launched agents cannot continue execution after permissions are revoked.

### Feature: Agent Swarm (Orchestrator + Workers)

**US-08**
> As a yield farmer, I want the agent swarm to execute deposits to all vaults in parallel once I approve their skills and grant permissions, so that I do not have to perform manual interventions at every step.

**Acceptance Criteria:**
- [ ] Once permissions are active + clicking "Launch Swarm": the Orchestrator dispatches all Workers in parallel.
- [ ] The vis.js graph displays the Orchestrator at the center and N Worker Agents.
- [ ] Worker Agents begin execution simultaneously (visible on the graph: nodes transition to the "running" state color).
- [ ] Each Worker Agent executes: Swap → Approve → Deposit without additional user interaction.
- [ ] If a single Worker fails: other Workers continue running (no abort).

**US-09**
> As a yield farmer, I want gasless execution, so that I can try multi-vault farming without paying any transaction fees.

**Acceptance Criteria:**
- [ ] All transactions are routed via the 1Shot Permissionless Relayer.
- [ ] No MetaMask prompt for gas confirmation is presented to the user during execution.
- [ ] Sepolia Etherscan displays that `from` = 1Shot relayer address, not the user's wallet.

**US-10**
> As a yield farmer, I want to be alerted if an agent attempts to perform actions outside of its granted permissions, so that I feel secure.

**Acceptance Criteria:**
- [ ] The smart contract triggers a revert (never fails silently) if the amount exceeds `maxAmount`.
- [ ] vis.js graph: the failed agent node turns red.
- [ ] The details panel displays a clear error message: "Execution reverted: amount exceeds permission".
- [ ] Other agents remain unaffected by the failure of a single agent.

---

### Feature: Agent Memory System

**US-11**
> As a yield farmer, I want to see what each agent has executed, the actual slippage, and the execution time, so that I have complete visibility over the agents' activities.

**Acceptance Criteria:**
- [ ] Click an agent node on the vis.js graph → details panel appears.
- [ ] The panel displays: executed step, status, `gasUsed`, `slippageActual`, and `executionTimeMs`.
- [ ] The panel displays the "lesson" recorded by the agent (if any).
- [ ] Memory is saved to `agents/memory/agent-{n}-memory.json` post-execution.

**US-12**
> As a yield farmer, I want agents to learn from their previous execution experiences, so that their performance improves over time.

**Acceptance Criteria (Could Have):**
- [ ] Memory from previous sessions is fed into the Venice AI prompt as context.
- [ ] Venice AI notes: "Based on previous executions, Vault A is reliable with 0.5% slippage."

---

### Feature: vis.js Graph Monitoring

**US-13**
> As a yield farmer, I want to view the status of all agents visually and in real-time, so that I know what is happening without having to read logs.

**Acceptance Criteria:**
- [ ] The vis.js Network graph is displayed once "Launch Swarm" is clicked.
- [ ] The Orchestrator node is shown in the middle, Worker nodes surrounding it, and Vault nodes on the outer ring.
- [ ] Clear edges are visible between Orchestrator→Worker and Worker→Vault.
- [ ] Node colors change in real-time: grey → blue → green / red.
- [ ] Updates occur when on-chain events are received (without page refresh).
- [ ] Clicking a node → details panel appears displaying the agent's information.

## 3. Non-Functional Notes

| Aspect | Requirement |
|-------|-------------|
| Privacy | Venice AI does not store user data (no data retention) |
| Security | Scoped permissions per-agent — no full wallet access |
| UX | Maximum of 8 clicks from landing page to agent swarm launch |
| Performance | Venice AI response time ≤ 10 seconds; vis.js graph updates ≤ 1 second after event |
| Reliability | A single Worker Agent failure does not abort other Worker Agents (Promise.allSettled) |
| Testability | Full flow can be demonstrated on Sepolia testnet with ≥ 2 parallel Workers |
| Parallelism | At least 2 Worker Agents run concurrently (not sequentially) |
